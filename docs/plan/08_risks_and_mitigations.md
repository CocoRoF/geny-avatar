# 08 — Risks & Mitigations

무엇이 프로젝트를 망가뜨릴 수 있는가, 어떻게 막을 것인가. 발생 가능성·영향 둘 다 적는다.

## R1 — 라이선스 (낮음 / hobby 단계 비차단)

**현 단계**: 1인 hobby 운영 컨텍스트라 Live2D Sample EULA의 General User, Spine 런타임의 evaluation 범주에 자연 통과. 차단 요인 아님.

**미래 트리거**: 공유 갤러리 / 결제 / 상업 배포로 스코프가 커지는 시점. 그때 [analysis/09 L1~L4](../analysis/09_open_questions.md) 재검증.

**완화 (preventative)**:
- 빌드인 데모 자산은 권리가 깨끗한 것만 사용 (Live2D 공식, Inochi2D 공식, 자체 제작) — 미래에 스코프가 커져도 빌드인 자산이 통째로 폐기되지는 않게.
- AI 결과·생성 메타데이터는 항상 사용자 브라우저에만 — 우리 서버에 영구 저장하지 않음 (D8).
- 자산 출처는 Avatar 메타에 가볍게 기록 — 미래에 라이선스 메타로 확장하기 쉽게.

## R2 — Atlas 분해가 너무 어려워 사용자가 견디지 못한다 (중 / 결정적)

**시나리오**: baked-in detail이 흔하다는 건 사실인데, SAM 자동 마스크가 부정확하고 사용자가 매번 브러시로 다듬어야 한다면 "AI로 다시 그려줘"가 매번 5분 작업이 됨.

**영향**: 핵심 가치 제안 무력화. 사용자는 "그냥 Photoshop 쓰겠다"가 됨.

**완화**:
- [analysis/05 접근 E (Reference-aware AI 재생성)](../analysis/05_texture_atlas_decomposition.md): 분해를 포기하고 region 통째로 다시 그리는 흐름을 default로. baked detail이 있어도 inpaint가 자연스럽게 새로 그림.
- "Auto" 가 부족하면 즉시 Generate로 fallback, 마스크 다듬기는 power user 모드.
- 내장 샘플은 처음부터 깨끗한 자산만 (분해 부담 적은 것).

## R3 — AI 결과 품질이 기대에 못 미친다 (중 / 큼)

**시나리오**: 30초 기다려서 받은 결과가 외곽선 어긋나거나 캐릭터 톤과 어긋남. 사용자가 5번 재생성해도 안 됨.

**영향**: 사용자 신뢰 하락. 비용 누적.

**완화**:
- ControlNet-canny weight 1.0으로 외곽선 강제. 후처리에서 silhouette mask로 alpha 강제 0.
- IP-Adapter + 캐릭터 LoRA 적층으로 일관성 강화 (Phase 5).
- 회귀 테스트 셋: 매 Phase 끝에 10 샘플 × 5 프롬프트 결과를 사람이 평가, 점수 내려가면 워크플로 롤백.
- 사용자에게 "결과가 부족하면 워크플로 변경" 옵션 노출 (anime 톤 base vs realistic 톤 등).

## R4 — Spine과 Live2D 동시 지원 비용 (수용한 비용)

**컨텍스트**: P1 ([README](../README.md))로 두 런타임을 V1에 같이 넣기로 결정했다. 이건 위험이 아니라 **의도적으로 받아들인 비용**이다.

**완화**:
- **어댑터 인터페이스를 처음부터 두 런타임으로 검증**: Phase 0 PoC에서 두 런타임 모두 띄워서 인터페이스 모양을 확정. 한 포맷만 보고 추상화한 인터페이스는 두 번째에서 깨진다는 사실을 첫날 마주하는 게 후반에 마주하는 것보다 싸다.
- **Capability flag**: 어댑터의 `capabilities.{canChangeMesh, canSwapTexture, canTint, ...}`로 한쪽만 되는 기능은 UI에서 회색 처리. 코어 로직은 분기 없이 capability 검사 한 번.
- **V1은 두 런타임이 모두 자신 있는 능력만**: 텍스처 region swap, slot/drawable visibility, color tint, 애니메이션 재생. mesh deform·새 attachment 추가 같은 비대칭 능력은 V2 이후.
- **테스트 자산은 두 포맷 동시 검증**: 회귀 테스트 셋에 Spine 모델과 Live2D 모델을 같이 넣어서, 한쪽만 통과하면 빌드를 실패로 처리.

## R5 — Replicate 비용이 폭증 (낮음 / 중)

**시나리오**: 사용자 늘어남 + 한 사용자가 시도/재생성 10번씩 → 월 $X 발생.

**영향**: 운영 부담.

**완화**:
- 개인 베타 단계까지는 일정 (Replicate Free + 자체 결제).
- 사용자 1인당 일일 quota (V1: 50회 / 일 등). 초과시 안내.
- 캐시 hit 강화 — 같은 입력은 재생성 안 함.
- V2에서 자체 ComfyUI 또는 BYO endpoint로 비용 분산.

## R6 — Cubism Core wasm 로딩 지연 (중 / 작음)

**시나리오**: Live2D 모델 띄우는 데 Cubism Core 로딩 + 모델 파싱이 합쳐서 3초+. 첫 페인트 1.5s 목표 위반.

**완화**:
- Cubism Core를 정적으로 prefetch.
- 첫 페인트 자산은 Spine 모델로 우선 호스팅 (Cubism Core 로딩 비용 회피). Cubism 모델은 사용자가 클릭한 후 lazy load.
- Worker에서 파싱.

## R7 — 1인 개발자 burnout (높음 / 큼)

**시나리오**: 6개월 짜리 18주 로드맵을 혼자 끌고 가다가 Phase 4 즈음에 동력 소진.

**완화**:
- Phase 1~3까지의 산출물(라이브 미리보기 + AI 생성)은 그 자체로 데모 가능 — 사용자 피드백을 일찍 받아 동기 보충.
- 작업 단위는 PR 단위 (continuous PR cadence — 사용자 메모리에 명시된 선호).
- 라이선스 결과(R1)가 회색지대면 차라리 V1 스코프를 줄여서 V0.5로 빨리 끝낸다.

## R8 — 의도치 않은 콘텐츠 생성 (낮음 / hobby 단계)

**현 단계**: 사용자 = 우리 자신. 외부 유포 없음. SaaS endpoint 자체 필터(Replicate)만 켜두면 충분.

**미래 트리거**: 공유 갤러리 또는 다른 사람이 우리 도구를 사용하기 시작하는 시점. 그때 단어 차단 + 동의 모달 + 더 강한 필터 추가.

## R9 — 기술 부채가 V1 끝에 폭발 (중 / 중)

**시나리오**: "동작하면 OK"로 만든 코드가 Phase 5 즈음에 도저히 못 굴릴 정도가 됨.

**완화**:
- Phase 끝마다 1~2일을 "구조 정비"에 할애 (refactor sprint).
- 어댑터 인터페이스를 처음부터 진지하게 — 첫 어댑터 결정 후 두 번째 어댑터를 임시로라도 한 번 만들어보면 인터페이스가 안정됨.
- 테스트는 V1 직전이 아니라 어댑터 단위 / atlas IO 단위로 미리미리.

## R10 — 사용자가 우리 도구의 출력을 신뢰하지 않는다 (중 / 큼)

**시나리오**: AI가 만든 텍스처가 "AI 느낌" 그대로라서 사용자가 자기 캐릭터 작품에 안 씀.

**완화**:
- 캐릭터 LoRA 지원 (Phase 5)으로 사용자 톤 강제.
- "Touch up" 기능 — 결과 위에 사용자가 브러시로 직접 다듬는 것 (Phase 6+).
- export 시 "AI 생성 비율 X%" 공시 — 투명성으로 신뢰 회복.

## R11 — 사용자 업로드 자산이 우리 런타임 버전 범위 밖 (중 / 중)

**시나리오**: 인터넷에서 받은 자산이 Spine 3.7 또는 Cubism 1 같은 우리가 받지 못하는 버전. 또는 비표준 패킹(파일 빠짐, 텍스처 경로 어긋남)으로 로딩 실패.

**영향**: 사용자가 "내 자산은 안 되네" 하고 도구를 떠남. P2 (업로드 day-1) 약속 위반.

**완화**:
- **버전 매트릭스 명시**: 받는 버전(Spine 3.8/4.0/4.1/4.2, Cubism 4/5 + best-effort 2/3)을 UI에 명확히 표기.
- **감지 → 명확한 오류 → 안내**: 받지 못하는 버전이면 "이 자산은 X 버전입니다. Y 도구로 변환하세요" 같은 구체적 안내. 침묵하지 않는다.
- **자산 다양성 회귀 테스트**: 인터넷에서 가져온 5종+ 자산을 Phase 1 종료 시점에 일괄 테스트. 80% 통과가 V1 수락 기준 ([plan/01 성공 측정](01_north_star.md)).
- **포맷 정상화 도구 안내**: 사용자가 자산을 마이그레이션할 수 있는 외부 도구(Spine Editor의 Export Old Version, Cubism Editor의 Reload as new format)를 도움말에 링크.

## 모니터링 — 어떻게 알 것인가

| 지표 | 어떻게 측정 | 임계 |
|---|---|---|
| AI 생성 실패율 | /api/ai 로그 | > 10% → 워크플로 점검 |
| 평균 응답 시간 | 같음 | > 30s → 백엔드 점검 |
| Replicate 월 비용 | Replicate dashboard | budget 임계 → 사용자 quota 강화 |
| 사용자 첫 세션 이탈 시점 | 우리 분석 (V1 이후) | 첫 generate 도달률 < 50% → UX 점검 |
| undo 사용 빈도 | store 로그 | 비정상 → UX 혼란 신호 |
