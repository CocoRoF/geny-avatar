## 변경 surface

### `components/LayersPanel.tsx`

`DecomposeStudio` (1380 LOC) + `GeneratePanel` (2633 LOC) 의 정적 import 제거. `next/dynamic` + `ssr:false` 로 래핑.

```ts
const DecomposeStudio = dynamic(
  () => import("@/components/DecomposeStudio").then((m) => m.DecomposeStudio),
  { ssr: false },
);
const GeneratePanel = dynamic(
  () => import("@/components/GeneratePanel").then((m) => m.GeneratePanel),
  { ssr: false },
);
```

두 컴포넌트는 이미 `{studioLayer && ...}` / `{generateLayer && ...}` 로 조건부 렌더링이라 ssr-skip 안전. 정적 import 만 잘라도 첫 페인트 시 4000+ LOC 가 빠진다.

### `components/ExportButton.tsx`

`buildExportZip` / `buildModelZip` 의 정적 import 제거. 핸들러 안에서 `await import(...)` 로 lazy load:

```ts
async function handleSave() {
  ...
  const { buildExportZip } = await import("@/lib/export/buildBundle");
  ...
}
```

fflate (~28KB) + zip 빌더 의존성이 사용자가 save / export 를 누를 때까지 다운로드되지 않는다. 버튼 자체는 즉시 표시.

### `app/layout.tsx`

```diff
- <Script src="/runtime/live2dcubismcore.min.js" strategy="beforeInteractive" />
+ <Script src="/runtime/live2dcubismcore.min.js" strategy="afterInteractive" />
```

`beforeInteractive` 는 FCP 직전 동기 로드 → first paint 까지 그 스크립트를 기다림. `afterInteractive` 는 페이지 interactive 직후 비동기 로드 → first paint 안 막음. `Live2DAdapter.waitForCubismCore()` 가 5s 폴링으로 already-handles 글로벌 부재. `lazyOnload` 는 idle 까지 미루는데, /edit/builtin/hiyori 직접 진입 시 5s 타임아웃 마진이 위태로워 안전한 `afterInteractive` 선택.

## 의도적 한계

- **adapter (Spine / Live2D) lazy split 안 함**: `AvatarRegistry.createAdapter` 가 동기. async 화 ripple 가 큰데 양쪽 모두 Pixi 의존 → 어차피 같은 chunk 로 떨어질 가능성. 측정 후 후속.
- **Pixi 자체 split 안 함**: Pixi v8 가 ~150KB. PuppetCanvas 가 시작부터 필요해서 미루기 어렵다. lazy-loading 시 빈 캔버스 깜빡임 발생.
- **HelpModal lazy 안 함**: 233 LOC 라 절대 이득 미미. `?` 키 핸들러가 정적 import 라 빠지면 closed-then-empty 모달 전환이 빈약.
- **첫 페인트 1.5s 측정 도구 안 깜**: Lighthouse / Vercel Web Vitals 셋업은 deploy 단계와 같이. 일단 build 사이즈 + 체감 + DevTools profiler.
- **runtime smoke test 안 함**: build 통과 + 정적 ssr:false 분리의 typing 안전. 사용자 시각 검증 필요 (특히 /edit/builtin/hiyori 의 Cubism 첫 mount).
