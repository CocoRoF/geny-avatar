# 04 — 전신 레퍼런스 리스타일 (G3)

목표: "레퍼런스 이미지를 주면, 개별 part 를 하나씩 도는 게 아니라 **전체 텍스처를
한 번에, 일관되게** 바꾼다." 두 개의 축으로 만든다:

1. **기반공사 — 페이지 오버라이드** (Stage 4): 아틀라스 페이지 전체를 교체 가능한
   1급 데이터로 만든다. 에디터(03/P-6)와 AI(아래) 모두 이 위에 선다.
2. **전신 리스타일 파이프라인** (Stage 5): 레퍼런스 + 전신 스냅샷을 조건으로
   페이지 단위 AI 변환을 오케스트레이션한다.

## 4-A. 페이지 오버라이드 기반공사

감사 결과 확인된 사실: 합성 메커니즘은 이미 페이지 단위다. 빠진 것은 (1) 페이지를
키로 하는 데이터 채널, (2) pristine 하드코딩 해제, (3) 영속/베이크/복원 배선.

### 키 규약
- **영속 키 = `pageIndex`** (model3.json `Textures[]` / .atlas 페이지 순서 — 로드 간 안정).
  `TextureId` 는 로드마다 재발급되므로 영속 금지 (Layer.id ↔ externalId 패턴과 동일).

### 변경 목록 (전부 additive — 하위 호환 유지)
| 지점 | 변경 |
|---|---|
| lib/store/editor.ts | `pageTextureOverrides: Record<number, Blob>` + setter. `setAvatar` 시 클리어 |
| lib/persistence (Option A) | `layerOverrides.kind` 에 `"pageTexture"` 추가, `layerExternalId = "page:<index>"` 인코딩. **Dexie 버전 bump 불필요**, 기존 인덱스 그대로 활용 |
| useLayerOverridesPersistence | 세 번째 diff 채널 (pageIndex 는 이미 안정 키라 번역 단계 없음) |
| lib/adapters/applyOverrides.ts | `pageBases?: Map<pageIndex, ImageBitmap>` 입력 — 페이지 재구성의 베이스를 pristine 대신 이것으로. **per-layer override/mask 는 그 위에 그대로 합성** (두 기능이 충돌 아닌 공존) |

### 선행 조건
- R1/R2 (applyOverrides 직렬화 + dirty-page 한정 재구성) — 같은 함수의 재설계라 먼저.
- D1 (deletePuppet 캐스케이드) — 페이지 PNG 는 지금까지 중 가장 큰 blob 클래스.

## 4-B. 전신 리스타일 파이프라인

### 왜 "페이지 통짜 AI 변환"이 1차 전략인가
- 아틀라스 페이지는 이미 모든 part 의 텍스처가 한 장에 모인 이미지다. 페이지를
  한 번에 변환하면 part 간 일관성이 **공짜로** 확보된다 (per-part 순차 생성의
  anchor/palette 체이닝 — docs-upgrade Phase 3 설계 — 보다 훨씬 단순).
- gpt-image-2 의 multi-image edits 는 "image[1]=아틀라스 페이지, image[2]=전신
  스냅샷(공간 컨텍스트), image[3..]=레퍼런스" 구도를 이미 지원한다. 기존
  per-layer 호출과 같은 API, 입력만 페이지로 확대.
- 실패 모드(부위 hallucination, 실루엣 붕괴)는 기존 postprocess 자산(alpha-enforce,
  erosion, 클라이언트 blend)을 페이지 치수로 일반화해 방어한다. **아틀라스 페이지의
  알파 실루엣은 절대 불변** — 결과를 원본 알파로 강제하면 UV 매핑이 깨질 수 없다.

### 흐름 (UI: GeneratePanel 과 별도의 "Restyle" 패널)
1. 사용자: 레퍼런스 이미지 선택(기존 ReferencesPanel 재사용) + 전역 프롬프트 입력.
2. 준비: 페이지별 현재 합성 이미지 (pageBase + layer overrides 베이크 = bakeAtlas
   로직 재사용) + `renderPuppetReference` 전신 스냅샷.
3. (옵션, 기본 on) refine: 기존 refine-prompt 라우트에 "이 이미지는 캐릭터의 텍스처
   아틀라스이고 영역별로 part 가 흩어져 있다, 레퍼런스의 스타일을 각 영역에 어떻게
   적용할지 영역 좌표 언급 없이 서술하라" 시스템 가이드 추가.
4. 페이지별 제출 (페이지 수는 보통 1–3장): 기존 `/api/ai/generate` 그대로, source =
   페이지 PNG. 페이지가 1024 초과면 제출 전 다운스케일, 결과를 원치수 업스케일
   (postprocess 의 기존 리샘플 경로 재사용) — gpt-image 출력 한계 대응.
5. postprocess: **원본 페이지 알파로 alpha-enforce** (erosion 은 페이지에선 0–2px 로
   약하게) → 결과를 페이지별 미리보기로 표시.
6. 사용자 확인 후 적용: `pageTextureOverrides[pageIndex] = blob` → 라이브 반영.
   페이지별 개별 revert 가능. 적용 전 단계에서 페이지별 재생성 가능.
7. provenance: aiJobs 히스토리에 `regionSignature = "page:<index>"` 로 기록.

### 품질 리스크와 대응
| 리스크 | 대응 |
|---|---|
| 페이지 전체 변환 시 모델이 아틀라스 구조를 "한 장의 그림"으로 오해 | 프롬프트에 atlas-sprite-sheet 명시 (falai composeInpaintingPrompt 의 "ISOLATED ATLAS TEXTURE REGION" 노하우 이식) + 전신 스냅샷을 공간 컨텍스트로 동봉 |
| 페이지 다운스케일로 디테일 손실 | 1차 출시는 수용 (단순함 우선). 추후 페이지를 사분면 타일로 분할 제출하는 고해상 경로 검토 |
| part 경계 침범 (어차피 알파 안에서만) | 알파 강제가 실루엣을 지키므로 渗み는 같은 part 영역 안 — 시각적 피해 제한적. 페이지별 blend 슬라이더(기존 mask-soft 로직 재사용)로 부분 복구 제공 |
| 페이지 간 스타일 불일치 (2페이지 이상 모델) | 페이지 1 결과를 페이지 2 제출의 ref 로 체이닝 (기존 lastResultBlob 패턴) |

### 명시적 비목표
- mesh/리깅 변경, 애니메이션 편집 — 기존 방침대로 스코프 외.
- per-drawable 순차 오케스트레이터 (docs-upgrade Phase 3 원안) — 페이지 통짜 방식이
  실패로 판명될 때의 fallback 으로 보류. router.ts 는 그때까지 dead code 로 두지 않고
  Stage 3 에서 제거 (필요 시 git 히스토리에서 복원).
