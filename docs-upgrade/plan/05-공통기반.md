# 05 — 공통 기반 작업

Phase와 무관하게 옆에서 같이 굴려야 하는 인프라 / 도구. 한 Phase 안에
다 욱여넣을 게 아니라 모든 Phase가 의존하는 공통 토대로 따로 관리한다.

대부분 **Phase 1 시작과 동시에 착수**하고, Phase 진행 내내 계속
갱신한다.

## 1 — Provider 추상화 정비

**왜**: 새 provider (FLUX.2)가 들어오고, 이후 라우팅 / 비용 추정 /
mask convention 변환 등이 모두 provider 계층에 깔끔히 들어가 있어야
함. 지금 `lib/ai/providers/`가 어떤 모양인지부터 점검.

**작업**:
- 기존 provider scaffold ([02-current-architecture.md](../02-current-architecture.md)
  참조) 그대로 따르되, 다음 메서드를 일급으로:
  - `submitGenerate(request)` — 핵심 호출.
  - `estimateCost(request)` — Phase 3 비용 표시용.
  - `maskConvention()` — alpha=0 = edit / alpha=1 = edit 중 어느
    것인지 명시.
  - `maxReferenceImages()` — image budget.
  - `supportsStreaming()` — Phase 3 progress UI 향상용.
- `lib/ai/providers/router.ts` — Phase 1 작업 5의 단순 라우터를
  Phase 3에서 발전. provider 우선순위 / fallback / cost-aware
  라우팅 모두 여기.

**점검 시점**:
- Phase 1 작업 4 (FLUX.2 추가) 들어가기 전에 한 번 정리.
- Phase 3 작업 4 (오케스트레이터) 들어가기 전에 한 번 점검.

## 2 — Worker 인프라

**왜**: 분류기 / baseline HSV / palette extraction / canny / tint
fallback 모두 메인 쓰레드에서 돌리면 UI 멈춤. worker 패턴을 일찍
표준화.

**작업**:
- `lib/workers/` 폴더 구조 정리. 이미 매직 셀렉터 / paint 등에 worker
  사용 중. 새 worker는 같은 패턴 따름:
  - 파일명 `.worker.js` (TypeScript 컴파일 이슈 회피).
  - 타입 정의 `.types.ts` 분리.
  - Comlink로 wrapping (또는 raw postMessage; 통일).
- 등록 / lifecycle 관리 유틸: `lib/workers/manager.ts` (필요 시).

**점검 시점**:
- Phase 2 작업 2 (분류기) 시작 전에 패턴 결정.

## 3 — Eval 러너

**왜**: 매 Phase의 ship criteria를 정량 측정하려면 자동 eval 인프라가
필수. 매번 손으로 확인하면 회귀가 안 잡힘.

**작업**:
- `eval/` 폴더 신설.
- `eval/runner.ts` — 메인 진입점. CLI로 실행 (`pnpm eval --phase=N`).
- `eval/groundtruth/` — 손으로 라벨링한 ground truth (분류기용).
- `eval/golden/intent.json` — 50-prompt intent eval set.
- `eval/reports/` — 실행마다 timestamp 폴더에 결과.

**Phase별 eval**:
- Phase 1: mask erosion 후 seam 픽셀 비율 측정.
- Phase 2: 분류기 F1, tint hue 정확도.
- Phase 3: intent parser 통과율, palette 분산.
- Phase 4: pixel mapping 정확도, animation seam.

**구현 순서**:
- Phase 1 진행 중에 runner 골격만. 첫 metric (seam ratio) 측정 가능
  수준.
- Phase 2 들어가기 전에 ground truth 라벨링 작업 분리.
- Phase 3 들어가기 전에 intent eval set 작성.

## 4 — Provenance / history 정합성

**왜**: 모든 Phase가 `editProvenance`에 의존. 이게 깨지면 audit /
undo / lock 정책 다 무너짐.

**작업**:
- Phase 2 작업 12에서 처음 등장. 모든 commit 진입점에서 호출되도록
  invariant 유지.
- 단위 테스트: 각 edit type별로 provenance가 정확히 1개 쓰이는지.
- 회귀 테스트: undo가 provenance entry를 reverse traversal하는 게
  올바른지.

## 5 — Live2D SDK 의존 점검

**왜**: pixi-live2d-display / Cubism SDK의 미공개 / 변동성 큰 API에
점차 의존이 늘어남. 어디까지 정상 동작하는지 정기적으로 확인.

**점검 항목** (실제로 코드 들여다보고 확인할 것):
- `renderToCanvas` 메서드 존재 / 안정성.
- `drawables.multiplyColors` slice 직접 쓰기 가능한지.
- mesh triangle / UV 좌표 노출 API.
- `setLayerTextureOverride` 의 동작 한계 (해상도 / 메모리).

각 [VERIFY] 태그가 [분석 문서들](..)에 있음. Phase
진입 직전 매번 검증하고 [VERIFY] 태그 제거 / 메모.

## 6 — 비용 / 토큰 가드레일

**왜**: AI 호출이 본격적으로 늘어나면 실수로 $50/day 비용이 흐를 수
있음. 모든 호출에 가드.

**작업**:
- session 단위 캡 — default $5. localStorage에 저장.
- 호출 직전 누적 비용 체크. 초과 시 명시적 모달.
- generate panel / orchestrator / dispatcher 모든 진입점에서 같은
  guard 호출.

**구현 시점**:
- Phase 1에서 단순 카운터 (호출 수)만.
- Phase 3에서 cost 추정과 묶어 본격 가드레일.

## 7 — 빌드 / 배포 점검

**왜**: 새 의존 / worker / IDB 마이그레이션이 빌드 / 배포에서 깨지는
경우가 자주 나옴. 매 Phase 마무리 시 점검.

**작업**:
- worker .js 파일이 build output에 포함되는지 확인 (`next.config.ts`).
- IDB 마이그레이션이 build 시점 / runtime 둘 다에서 동작.
- IDB 용량 추정치를 dev 모드에서 콘솔에 띄움 (개발자 경각심).
- Docker / Vercel / VSCode extension 빌드 모두에서 smoke test.

**점검 시점**:
- 매 Phase 종료 직전.

## 8 — 문서 / 자취 관리

**왜**: docs-upgrade 분석은 Phase 진행하면서 점점 stale해진다. 변경
지점을 plan에서 docs-upgrade로 역feedback해야 함.

**규칙**:
- Phase 종료 시 진행기록 ([06-진행기록.md](06-진행기록.md))에 결정 사항
  / 변경점 적기.
- docs-upgrade 문서가 결정과 어긋나면 해당 문서 업데이트. plan은
  실행 의도, docs-upgrade는 시점 시점의 결정 record.
- 분석 폐기 시 docs-upgrade 문서에 `**REVISED**: <이유>` 헤더만
  남기고 본문 유지 (history 보존).

## 9 — 솔로 작업 페이싱

**왜**: 취미 / 솔로 작업이므로 burn-out 방지가 ship 보다 우선.

**규칙**:
- 한 Phase 당 PR은 5–10개 정도. 너무 큰 PR 만들지 말 것.
- 각 PR은 독립적으로 reviewable / revertable. 의존 PR은 명시.
- 매 Phase 종료 직후 일주일 정도 idle / polish 기간 (다음 Phase
  설계 재검토).
- 사용자(나) 입장에서 "이거 진짜 필요한가?" 질문을 주기적으로.

## 10 — Phase 5 (PSD round-trip) 트래킹

**왜**: Phase 5는 defer지만 정보는 계속 모음. `.moc3` write 커뮤니티
도구가 갑자기 좋아질 수 있고, 사용자가 PSD source 들고 들어오는
빈도가 변할 수 있음.

**작업** (분기별):
- moc3ingbird / OpenL2D / Inochi 진척도 점검.
- 사용자 telemetry (구현되면) 에서 PSD source upload 비율 추정.
- 5% 넘으면 Phase 5 재검토.
