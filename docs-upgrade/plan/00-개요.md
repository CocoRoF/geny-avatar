# 00 — 계획 폴더 개요

`docs-upgrade/`의 분석 결과를 **내가 직접 손으로 칠 작업 목록**으로
바꿔 적어둔 폴더. 분석은 옆 폴더에 있고, 이 폴더는 "그래서 뭘 칠
거냐"만 다룬다.

geny-avatar는 **단독으로 동작하는 프로젝트**다. geny 본체에 의존하지
않는다. 모든 계획은 이 저장소 안에서 닫혀 있어야 한다.

## 폴더 구성

| 파일 | 다루는 범위 | 예상 기간 |
|---|---|---|
| `01-Phase1.md` | 레이어 파이프라인 정비 + FLUX.2 escape hatch | 2–3주 |
| `02-Phase2.md` | 시맨틱 그룹 + 틴트 fast-path | 4–6주 |
| `03-Phase3.md` | 인텐트 레이어 + 순차 AI 오케스트레이터 | 6–9주 |
| `04-Phase4.md` | 렌더-앤-백프로젝션 (spike 통과 시) | 1주 spike + 8–12주 |
| `05-공통기반.md` | 단계와 무관하게 같이 굴려야 하는 인프라 | 상시 |

실제 진행 로그는 옆 폴더 [`../progress/`](../progress/)에 작업 단위로 누적.

## 어떤 순서로 읽고 칠 거냐

1. **이 파일** — 원칙 / 글로벌 약속을 먼저 확인.
2. **`01-Phase1.md`** — 다음 주에 손에 잡을 작업. 진짜 시작점.
3. Phase 1이 끝나면 `02-Phase2.md`로 이동. 단계마다 그 안에서
   "작업 → 검증 → 커밋" 사이클을 돈다.
4. **`05-공통기반.md`**은 Phase 1 시작과 동시에 옆에서 돌아간다.
   provider 추가, IDB 마이그레이션 골격, eval 러너는 단계와
   무관하게 항상 갱신 대상.
5. 매 작업 단위로 [`../progress/`](../progress/)에 로그 파일을 한 개씩
   append. 양식과 운영 규칙은 [`../progress/README.md`](../progress/README.md).

## 글로벌 원칙 (모든 단계 공통)

이 5개를 어기는 PR은 머지하지 않는다.

1. **하위 호환은 깨지 않는다.** 기존 puppet 파일, IDB, 세션은
   업그레이드 후에도 그대로 열린다. 마이그레이션은 일방향이지만
   기존 데이터를 손상시키지 않는다.
2. **per-layer 파이프라인은 그대로 둔다.** Phase 2/3/4의 새 코드는
   결국 `submitGenerate` + `setLayerTextureOverride`를 호출한다.
   오케스트레이터는 그 위에 얹는 레이어지 대체가 아니다.
3. **provenance는 필수.** 모든 편집은 `editProvenance`에 기록한다.
   AI든 paint든 mask든 tint든.
4. **lock 플래그는 무조건 존중한다.** Phase 2 이후 모든 그룹 단위
   작업은 `groupLocked = true` 인 drawable을 건너뛴다.
5. **애니메이션 검증을 통과해야 커밋.** 매 단계의 편집 결과는
   6개 파라미터 극단값에서 렌더해보고 seam/drift 없는지 확인한
   뒤 user-visible commit이 가능하다.

## 사용 기술 / 외부 의존

추가로 들어올 새 의존성은 미리 적어둔다 (단계마다 PR로 들어옴):

- `fal-client` (Phase 1) — FLUX.2 호출용. `lib/ai/providers/falai.ts` 신설.
- `zod` (이미 있음) — intent parser 응답 검증.
- `comlink` 또는 native worker (Phase 2) — 분류기 / 히스토그램 계산용
  off-thread 처리. 이미 있는 worker 패턴 재사용.
- `color-rgba` / `culori` (Phase 2) — HSV ↔ multiplyColor 수학.
  외부 의존 추가 vs 직접 구현 결정은 Phase 2 첫 작업에서 정함.
- `pixi-live2d-display` (이미 있음) — canonical pose 렌더용
  `renderToCanvas` 활용. [VERIFY] 필요.

## "이번 PR에서 안 한다" 목록

플랜 외부 욕심으로 자꾸 들어오는 것들 — 의식적으로 거른다.

- 새 도메인 모델 만들기 (semantic group 외).
- 새 UI 프레임워크 도입 / 디자인 시스템 교체.
- 백엔드 서버 추가 (다 클라이언트 사이드로 마무리).
- 결제/요금제 UI. 비용 estimate는 표시만 하고 결제 흐름은 별개.
- 모바일 대응. 데스크탑 브라우저만 1차 타깃.

## 시간 가정

문서들에서 적힌 "주" 단위는 **취미 / 솔로 / 비집중 모드** 기준이다.
풀타임 집중 시 절반으로 짧아질 수 있고, 주말만 작업하면 두 배가 될
수 있다. 측정 단위로 본다기보다 작업의 **상대적 무게**로 본다:
Phase 1은 Phase 2의 절반 이하, Phase 3은 Phase 1+2 합보다 무겁다.

## 의사결정 지점

각 Phase가 끝날 때마다 다음 Phase로 들어가기 전에 잠깐 멈춘다:

- 직전 Phase의 ship criteria가 진짜 만족됐나? 아니면 다음으로
  가지 말고 한 번 더 polish.
- 직전 Phase에서 배운 게 다음 Phase 설계를 바꾸나? 바꾼다면
  해당 `docs-upgrade/` 문서를 같이 업데이트.
- 여전히 이 방향이 맞나? 유저(나) 입장에서 헛수고면 다른 트랙
  잡는다.

플랜은 트레드밀이 아니다. 멈출 권리는 매 단계마다 있다.
